> - HTTP 실패 3%는 클라이언트 timeout(10초) 초과 — 요청이 큐에서 10초 이상 대기 후 클라이언트가 먼저 끊은 것이다.
> - **메모리 누수 없음 확인**: 30분간 RPS가 3.94로 일정하게 유지 — 시간에 따른 latency 누적 악화나 처리량 감소가 없으므로 메모리 누수·연결 고갈은 없다고 판단한다.

#### Worker 분리가 실제로 효과적인 시나리오

```
분리 전 (Phase 3A):
  [/predict × 10 VU]  ─┐
                        ├── 0.5 vCPU 공유 → 재학습 중 추론 latency 급등
  [/model/retrain]    ─┘

분리 후 (Phase 3B):
  [/predict × 10 VU]  ──→ API Task 0.5 vCPU (추론 전용)
  [/model/retrain]    ──→ SQS 메시지 발행 후 즉시 반환
                          → Worker Task 1.0 vCPU (학습 전용, 비동기)
```

> 이 테스트에서는 재학습 트리거가 없었으므로 Worker 분리의 핵심 효과(재학습 중 추론 latency 보호)는 수치에 반영되지 않았다.

## 3. 단계별 개선량 분석

### Phase 1 → Phase 2: 로컬 수평 확장

**변경 내용:** 단일 컨테이너 → 컨테이너 2개 + nginx 라운드로빈

| 지표 | Phase 1 | Phase 2 | 변화량 | **개선율** |
|------|---------|---------|--------|---------|
| Baseline p50 | 818 ms | 255 ms | -563 ms | **▼ 68.8%** |
| Baseline p95 | 2,979 ms | 2,254 ms | -725 ms | **▼ 24.3%** |
| Baseline p99 | 8,074 ms | 4,182 ms | -3,892 ms | **▼ 48.2%** |
| Baseline RPS | 4.61 | 6.41 | +1.80 | **▲ 39.0%** |
| Baseline 총 요청 수 | 1,384 | 1,922 | +538 | **▲ 38.9%** |
| Baseline HTTP 실패율 | 0.7% | 0.0% | -0.7%p | **▼ 100%** |
| Spike 오류율 | 81.9% | 67.3% | -14.6%p | **▼ 17.8%** |
| 데이터 일관성 (miss rate) | N/A | **100%** | — | 신규 문제 발견 |

**효과 요약:**
- p50 latency가 68.8% 감소 — 2개 컨테이너가 병렬 처리하여 큐 대기 시간 절반 이하로

- RPS 39% 향상 — 처리량 증가

## 4. CPU가 진짜 병목이었는가

### 결론: 예, CPU BERT 추론이 전 구간에서 일관된 병목이었다

#### 근거 1: Spike 테스트의 15,000ms 수렴 패턴

모든 Phase에서 Spike(100 VU) 시 p50/p95가 동일하게 **~15,000ms**에 수렴한다.

| Phase | Spike p50 | Spike p95 |
|-------|-----------|-----------|
| Phase 1 | 14,999 ms | 15,016 ms |
| Phase 2 | 14,998 ms | 15,005 ms |
| Phase 3A | 15,001 ms | 15,001 ms |
| Phase 3B | 15,001 ms | 15,001 ms |

> 이 숫자는 k6 스크립트의 클라이언트 timeout 값(15초)이다. 서버가 응답을 못 보내는 것이 아니라, **큐에서 대기 중 클라이언트 timeout이 먼저 걸리는 것**이다. 이는 CPU가 100VU 동시 요청을 소화하지 못해 요청이 큐에 쌓이는 현상이다.

#### 근거 2: 컨테이너 수를 2배로 늘리자 p50이 정확히 절반 가까이 감소

Phase 1(1개) → Phase 2(2개)에서 p50이 818ms → 255ms로 감소. 이는 큐 대기 시간이 줄어든 것으로, 각 컨테이너가 받는 동시 요청이 절반으로 줄어든 것이다. 이 패턴은 CPU 처리 병목에서 전형적으로 나타난다.

#### 근거 3: ECS vCPU 제한 → 로컬 대비 latency 급증

Phase 2(로컬, CPU 무제한)에서 Phase 3A(ECS 0.5 vCPU)로 전환하자 p50이 255ms → 1,711ms로 570% 증가. 네트워크 홉 overhead(50~100ms)로는 설명이 불가능한 수준이다. ECS의 0.5 vCPU 제한이 BERT 추론 속도를 직접적으로 늦춘 것이다.

#### 근거 4: Soak(30 VU)에서 처리량 변화 없음(RPS 3.94 일정), 하지만 latency 폭발

Phase 3B Soak에서 p95가 9,469ms임에도 RPS는 30분간 3.94로 일정하다. 이는:
- 서버가 죽거나 메모리 누수가 일어나는 것이 아님
- CPU가 처리할 수 있는 RPS 한계(약 4 req/s)에 도달, 나머지는 큐에서 대기
- 30 VU × 요청 빈도가 4 req/s를 초과하는 순간부터 큐가 계속 쌓임

#### CPU 병목 구조 요약

```
BERT 추론 처리 능력 (0.5 vCPU 기준):
  - 단일 추론 평균 시간: ~1,600ms
  - 동시 처리 가능 용량: ~1 req at a time (단일 쓰레드 추론)
  - 이론적 최대 RPS: 1 / 1.6s = 0.625 req/s × (병렬 처리 수)
  - 실측 RPS: ~3.76~3.94 req/s (FastAPI async + 큐잉 효과)

10 VU → 큐 대기 짧음 → p95 2,300~2,400ms
30 VU → 큐 대기 김 → p95 9,469ms
100 VU → 큐 폭발 → 15초 timeout
```

#### CPU 한계가 아닌 부분에서의 개선 효과

| 개선 사항 | Phase | CPU 관련 여부 | 결과 |
|---------|-------|------------|------|
| 컨테이너 2개 분산 | Phase 2 | CPU 분산 | p50 -68.8% |
| 클라우드 신뢰성 | Phase 3A | CPU 무관 | HTTP 실패 0%, 데이터 일관성 확보 |
| Worker 분리 | Phase 3B | CPU 격리 | 재학습 중 추론 CPU 보호 |
