# GPU 추론 기반 부하 테스트 예측 리포트

> **문서 성격:** 이 문서의 모든 수치는 실측값이 아닌 **예측값(Prediction)**이다.
> CPU 실측 데이터를 기반으로 GPU 추론 속도 차이를 반영하여 산출한 추정치이며,

## 예측 전제 조건 (Assumptions)

### GPU 사양 가정

| 항목 | 가정값 | 근거 |
|------|--------|------|
| GPU 모델 | NVIDIA T4 (g4dn.xlarge) | AWS EC2 추론 용도 표준 선택 |
| vCPU | 4 vCPU | g4dn.xlarge 스펙 |
| RAM | 16 GiB | g4dn.xlarge 스펙 |
| GPU VRAM | 16 GiB | T4 스펙 |
| 비용 (On-Demand) | ~$0.526/hr | ap-northeast-2 기준 |
| 비용 (Spot) | ~$0.158/hr | 약 70% 절약 |

### 모델 추론 속도 가정

| 환경 | 단일 추론 시간 | 최대 처리량 (추정) |
|------|-------------|----------------|
| CPU (0.5 vCPU, 실측) | ~1,500 ms | ~4 req/s |
| T4 GPU (batch_size=1) | **~12 ms** | ~280 req/s (단일 컨테이너) |
| **속도 향상 배수** | **~125배** | **~70배** |

> **추론 시간 산출 근거:**
> - snunlp/KR-FinBert-SC (BERT-base 아키텍처) batch_size=1 기준
> - T4 GPU에서 BERT-base 추론: 논문·벤치마크 기준 8~15ms (평균 12ms 사용)
> - Python FastAPI + PyTorch CUDA 오버헤드 포함

### 아키텍처 변경 사항

> **중요:** ECS Fargate는 GPU를 지원하지 않는다.
> Phase 3A/3B에서 GPU를 사용하려면 **ECS on EC2** (GPU 최적화 AMI) 구조로 변경이 필요하다.

| Phase | CPU 실측 인프라 | GPU 예측 인프라 |
|-------|--------------|--------------|
| Phase 1 | 단일 Docker (로컬) | 단일 Docker (로컬, --gpus all) |
| Phase 2 | Docker 2개 + nginx (로컬) | Docker 2개 + nginx (로컬, 각 GPU 연결) |
| Phase 3A | ECS **Fargate** 0.5 vCPU | ECS **on EC2** g4dn.xlarge (T4 GPU) |
| Phase 3B | ECS **Fargate** + Worker 분리 | ECS **on EC2** g4dn.xlarge + Worker 분리 |

### 네트워크 오버헤드 가정

| 구간 | 추가 latency |
|------|------------|
| 로컬 Docker (Phase 1/2) | 0 ms |
| API Gateway → ALB → ECS EC2 | +35 ms (실측 기반 추정) |

#### Worker 분리 + GPU 동시 재학습 시나리오 (추가 예측)

> CPU 환경에서는 이 시나리오를 측정하지 않았으나, GPU 환경에서의 예측값을 제시한다.

```
시나리오: 10 VU 추론 요청 + 동시에 /model/retrain 요청 발생

Phase 3A (GPU, Worker 미분리):
  API Task GPU: 추론 12ms + 재학습 BERT fine-tuning (~30s) CPU 공유
  → 재학습 중 GPU VRAM 경합 발생
  → 추론 p95: ~350ms (재학습 없을 때의 112ms × ~3배 예상)

Phase 3B (GPU, Worker 분리):
  API Task GPU:  추론 전용 → p95: 89ms (재학습 영향 없음)
  Worker Task:   재학습 전용 (별도 CPU/메모리) → 추론 latency 무영향
```

| 지표 | Phase 3A GPU (재학습 동시) | Phase 3B GPU (Worker 분리) | 개선율 |

## 4. GPU 환경에서의 병목 이동

### CPU vs GPU 병목 비교

```
[CPU 환경]
병목: BERT 추론 자체 (~1,500ms/req)
→ 10 VU만으로도 큐 대기 발생
→ 100 VU에서 완전히 포화

[GPU 환경]
병목: 서버 동시 연결 처리 수 (async worker 수)
→ 단일 컨테이너: ~280 req/s
→ 100 VU(~870 req/s 수요) 에서 큐 발생
→ 그러나 GPU가 빠르게 처리해 큐 해소 속도도 빠름
```

### GPU Spike 실패 원인 분석 (Phase 1)

```
Phase 1 GPU Spike 실패 원인:
  문제: 추론 속도(X) → 병목이 아님 (GPU 12ms)
  문제: async worker 수 제한 (Uvicorn 기본 1~4 worker)
          → 초당 처리 가능 요청 수: 4 workers × (1/0.012s) = 333 req/s
          → 100 VU 수요: ~870 req/s
          → 초과분 큐에 적재 → 일부 timeout

해결 방법 (Phase 2에서 자동 해결):
  컨테이너 2개 → 처리량 2배 (666 req/s) → 870 req/s 수요 대응 가능
```

### 병목 요약표

| Phase | CPU 병목 | GPU 병목 |
|-------|---------|---------|
| Phase 1 | BERT 추론 속도 (1,500ms) | async worker 수 제한 (단일 컨테이너) |
| Phase 2 | BERT 추론 속도 (1,500ms) | 없음 (560 req/s로 수요 감당) |
| Phase 3A | BERT 추론 속도 + 0.5 vCPU 제한 | 네트워크 홉 (~35ms) |
| Phase 3B | BERT 추론 속도 | 네트워크 홉 (~35ms) |

### GPU 전환 후 추가 확인 필요 사항

| 항목 | 내용 |
|------|------|
| **Cold Start** | ECS EC2 GPU 인스턴스 시작 시 모델 로드: ~30~60초 예상. ALB health check 통과 전 traffic 차단 확인 필요 |
| **GPU 메모리 관리** | BERT-base 모델 크기 ~440MB. T4 16GB VRAM에서 여러 모델 동시 로드 시 OOM 여부 확인 |
| **CUDA 버전 호환** | PyTorch CUDA 버전과 g4dn GPU Driver 호환 확인 (CUDA 11.x 또는 12.x) |
| **비용 증가** | CPU ECS Fargate 0.5 vCPU: ~$9/월 → g4dn.xlarge On-Demand: ~$378/월 (약 42배) |
| **ECS Fargate → EC2 전환** | Fargate GPU 미지원 → EC2 Autoscaling Group 관리 필요 (Fargate 대비 운영 복잡도 상승) |
| **배치 추론 최적화** | 현재 batch_size=1. GPU 활용도 극대화를 위해 Dynamic Batching 도입 시 처리량 추가 향상 가능 |

### 비용 대비 성능 비교

| 환경 | 월 비용 (추정) | Baseline p95 | Spike 오류율 | 비용 효율 |
|------|-------------|-------------|------------|---------|
| ECS Fargate 0.5 vCPU (CPU) | ~$9/월 | 2,328ms (FAIL) | 95.1% (FAIL) | 낮음 |
| ECS Fargate 1.0 vCPU (CPU 업그레이드) | ~$18/월 | ~1,200ms (예측) | ~60% (예측) | 중간 |
| ECS EC2 g4dn.xlarge (T4 GPU) | ~$378/월 | **89ms (PASS)** | **2.9% (PASS)** | 높음 (SLO 달성) |

> **권장:** Spot 인스턴스 활용 시 월 ~$114로 모든 SLO 통과 가능. 중단 허용 시간을 SQS + ECS 재배포로 처리하면 Spot 안정성 확보 가능.
