## 1. Stage 1 문제점 — 왜 고도화가 필요했는가

Stage 1은 단일 FastAPI 컨테이너 + 로컬 파일시스템 구조다.
서비스를 운영하는 순간 아래 5가지 문제가 동시에 발생한다.

### 문제 1: 단일 장애점 (Single Point of Failure)

컨테이너 하나가 유일한 서버다.
재시작·업데이트·OOM kill 어떤 상황에서도 서비스가 전면 중단된다.
오토스케일링도, 헬스체크 기반 자동 복구도 없다.

```
Stage 1 구조:
  사용자 → [FastAPI 컨테이너 1개] → 로컬 파일
             ↑ 여기 하나가 죽으면 전체 서비스 정지
```

### 문제 2: 로컬 파일시스템 — 데이터 소실 위험

학습 데이터(`user_data/training_data/*.jsonl`)와 모델 파일이 컨테이너 로컬 볼륨에 저장된다.
컨테이너를 재배포하거나 재시작하면 모든 학습 데이터가 영구 소실된다.
사용자가 누적한 피드백(라벨 데이터)이 통째로 날아간다.

### 문제 3: 수평 확장 불가 + 데이터 불일치

컨테이너를 2개로 늘리면(Stage 2) nginx가 요청을 분산하지만,
각 컨테이너가 **자신의 로컬 파일**에만 데이터를 저장한다.
사용자가 server_a에 피드백을 저장하고 server_b에 조회하면 데이터가 없다.

```
Stage 2 실측: 조회 1,845건 중 불일치 1,845건 → 불일치율 100%
```

스케일아웃을 하면 할수록 데이터 불일치가 증가하는 구조적 모순이 생긴다.

### 문제 4: 추론 + 재학습 CPU 경합

`/model/retrain`이 호출되면 BERT fine-tuning이 **동일 프로세스의 백그라운드 스레드**에서 실행된다.
추론 요청과 재학습이 같은 CPU를 공유하므로, 재학습 중에 추론 latency가 폭발적으로 증가한다.

```
같은 4 vCPU 조건 실측 결과:
  Stage 1 (백그라운드 스레드): 재학습 중 p95=866ms, 오류율=49.93% → FAIL
  Phase 3B (Worker 분리):      재학습 중 p95=107ms, 오류율=0.00%  → PASS
```

서비스 중에 모델을 개선할수록 서비스 품질이 떨어지는 역설이 발생한다.

### 문제 5: 운영 가시성 없음

로그가 컨테이너 stdout에만 남는다. 메트릭도, 알람도, 대시보드도 없다.
CPU 100%, 메모리 고갈, 오류 급증이 발생해도 누군가 직접 보기 전까지 알 수 없다.
서비스 장애 원인을 알아도 재현 가능한 배포(IaC) 체계가 없어 롤백이 수동이다.

### Stage 1 문제점 요약

| 문제 | 원인 | 실측 피해 |
|------|------|---------|
| 단일 장애점 | 컨테이너 1개 | 장애 시 전체 서비스 정지 |
| 데이터 소실 | 로컬 파일시스템 | 컨테이너 재시작 → 학습 데이터 전체 소실 |
| 스케일아웃 불가 | 로컬 상태 | 2개 컨테이너 시 데이터 불일치율 100% |
| CPU 경합 | 백그라운드 스레드 | 재학습 중 오류율 50%, p95 866ms |
| 운영 불가 | 관측성 없음 | 장애 감지 불가, 수동 배포·롤백 |

## 2. 인프라 선택 근거

### 2.1 왜 ECS Fargate인가 (EC2가 아닌 이유)

#### EC2의 문제점

EC2는 가상머신이다. 서버를 빌려서 그 위에 직접 운영체제, Docker, 모니터링 에이전트를 설치하고 관리해야 한다.

| 작업 | EC2 | ECS Fargate |
|------|-----|-------------|
| OS 패치 | 직접 실행 필요 | AWS가 처리 |
| Docker 설치·업그레이드 | 직접 | AWS가 처리 |
| 인스턴스 헬스체크 | 직접 구성 | ALB 연동으로 자동 |
| 스케일아웃 | Auto Scaling Group 별도 구성 | `desired_count` 하나 변경 |
| 비용 | 24시간 고정 (idle time 포함) | 태스크 실행 시간만 과금 |
| 배포 중단 없는 업데이트 | Blue/Green 직접 구성 | ECS rolling update 기본 제공 |

#### ECS Fargate를 선택한 이유

**1. 인프라 관리 제거**: OS, Docker 엔진, 인스턴스 수명주기 관리를 AWS에 위임한다.
코드와 컨테이너 이미지만 관리하면 된다.

**2. 태스크 단위 독립 스케일아웃**: API와 Worker를 각각 독립적인 Fargate 서비스로 분리할 수 있다.
`desired_count = 2`로 설정하면 API 태스크가 2개 병렬로 동작하고, ALB가 자동 분산한다.
Worker는 재학습 부하에 맞게 별도로 스케일 가능하다.

**3. Terraform으로 완전한 IaC**: 모든 인프라를 코드로 선언하고 `terraform apply` 한 번으로 재현 가능하다.
EC2도 Terraform으로 관리할 수 있지만, OS 수준 설정(패키지, 서비스 등)은 별도 도구(Ansible, user_data 스크립트)가 필요하다.
Fargate는 컨테이너 이미지만 지정하면 되므로 IaC 범위가 단순하다.

**4. 비용**: 프로토타입/소규모 서비스 기준, 0.5 vCPU / 1 GiB ECS 태스크는 월 약 $9~12 수준이다.
EC2 t3.small (2 vCPU, 2 GiB)은 월 약 $15~18이며, idle 시간에도 과금된다.

#### Lambda를 사용하지 않은 이유

Lambda는 서버리스로 Fargate보다 더 간단하지만, 이 서비스에는 맞지 않는다.

- **콜드 스타트**: BERT 모델 로드가 수 초 걸린다. Lambda 콜드 스타트마다 사용자가 초기 지연을 경험한다.
- **실행 시간 제한**: Lambda 최대 15분. BERT fine-tuning은 15~30분이 소요되므로 재학습에 사용 불가.
- **메모리 한계**: 모델 크기와 추론 메모리 요구량이 Lambda 한계(10 GiB)에 근접할 수 있다.
- **스트리밍 불가**: 실시간 YouTube 댓글 분류 응답을 스트리밍하기 어렵다.

ECS Fargate는 컨테이너를 상시 실행하는 "서버처럼" 동작하므로 모델이 메모리에 로드된 채로 대기하다가 즉시 응답한다.

### 2.2 왜 S3인가

#### 기존 구조의 문제

Stage 1/2에서 학습 데이터(`*.jsonl`)와 모델 파일이 컨테이너 로컬 볼륨에 있다.

- **데이터 소실**: ECS 태스크가 재시작되면 에피머럴 스토리지가 초기화된다.
- **데이터 불일치**: ECS 태스크가 2개라면 각자 다른 데이터를 갖는다.
- **EFS 대신 S3를 선택한 이유**: EFS(Elastic File System)도 여러 컨테이너가 공유 마운트 가능하지만, 지속적인 마운트 비용이 발생하고, BERT 모델처럼 대용량 바이너리 파일의 버전 관리가 어렵다.

#### S3를 선택한 이유와 효과

| 특성 | 효과 |
|------|------|
| 모든 ECS 태스크가 동일 버킷 접근 | 어떤 태스크가 응답해도 동일 데이터 반환 → **불일치율 0%** |
| 객체 영구 보존 | 태스크 재시작·재배포·스케일아웃 시에도 데이터 소실 없음 |
| 객체 키 기반 버전 관리 | `models/{version}/` 경로로 자동 모델 버전 관리 |
| 고가용성 (99.999999999% 내구성) | 별도 백업 불필요 |
| 비용 | 저장된 데이터 GB당 ~$0.025/월 (소규모 서비스 기준 $1 미만) |

```
S3 데이터 흐름:
  사용자 피드백 → API → S3 training-data/{date}.jsonl  (영구 보존)
  /model/retrain  → Worker → S3 training-data/ 다운로드
                           → BERT fine-tuning
                           → S3 models/{version}/ 업로드
  다음 API 기동   → S3 models/{latest}/ 다운로드 → 추론 서빙
```

### 2.3 왜 RDS PostgreSQL인가

#### S3만으로 부족한 이유

S3는 파일 저장소다. 아래 작업은 S3만으로는 할 수 없다.

- "가장 최근 재학습 실행이 성공했는가?" 조회
- "오늘 피드백이 몇 건 들어왔는가?" 집계
- 동시에 여러 API 태스크가 피드백을 저장할 때 트랜잭션 일관성 보장

#### RDS PostgreSQL을 선택한 이유

| 특성 | 효과 |
|------|------|
| ACID 트랜잭션 | 여러 ECS 태스크가 동시에 쓰기 요청을 보내도 데이터 충돌 없음 |
| SQL 집계·조회 | 재학습 이력, 피드백 통계, 라벨 분포를 즉시 조회 가능 |
| 관리형 서비스 | 자동 백업(7일), 멀티-AZ 장애조치, 패치 적용이 AWS 담당 |
| 비용 | db.t3.micro 기준 월 약 $15 |

**DynamoDB 대신 PostgreSQL을 선택한 이유**: DynamoDB는 Key-Value 접근에 최적화되어 있다. `training_runs` 테이블에서 "최근 성공한 재학습 버전 조회", "날짜 범위 필터" 등 SQL 쿼리가 필요한 작업에는 PostgreSQL이 훨씬 직관적이다. 이 규모에서 NoSQL의 확장성 이점은 필요 없다.

### 2.4 왜 SQS + Worker 분리인가

#### 백그라운드 스레드 방식의 문제

Stage 1의 `/model/retrain`은 FastAPI BackgroundTasks를 사용한다.
이 방식은 같은 프로세스 내에서 스레드로 재학습을 실행한다.

```
문제: 추론 요청과 재학습이 동일 프로세스(동일 CPU)를 공유

시나리오: 재학습 요청 1건 + 동시 추론 요청 10건
  → BERT fine-tuning이 vCPU 대부분을 점유
  → 추론 요청이 CPU를 할당받지 못해 대기
  → 결과: p95 급등, 오류율 증가
```

#### 같은 4 vCPU에서 실측 비교

| 구조 | p95 latency | 오류율 | 판정 |
|------|-------------|--------|------|
| 백그라운드 스레드 (Stage 1, 4 vCPU) | **866 ms** | **49.93%** | FAIL |
| SQS + Worker 분리 (Phase 3B, 4 vCPU) | **107 ms** | **0.00%** | PASS |

4 vCPU로 CPU를 아무리 늘려도 같은 프로세스에서 경합하면 오류율 50%가 발생한다.

#### SQS + Worker ECS를 선택한 이유와 효과

```
분리 후 아키텍처:

  사용자 → /model/retrain → API Task
                            └─ SQS 메시지 발행 (즉시 반환, ~5ms)
                                └─ Worker Task (독립 ECS 태스크)
                                   ├─ SQS 롱폴링 수신
                                   ├─ S3에서 학습 데이터 다운로드
                                   ├─ BERT fine-tuning (~15~20분)
                                   ├─ S3에 모델 업로드
                                   └─ RDS에 결과 기록
```

| 특성 | 효과 |
|------|------|
| CPU 격리 | API Task(추론 전용)와 Worker Task(재학습 전용)가 각자 독립 vCPU 사용 |
| 즉시 반환 | `/model/retrain` 호출 즉시 `200 OK` 반환 → 사용자 UX 향상 |
| 독립 스케일 | Worker만 CPU를 높이거나 여러 개 실행 가능 |
| DLQ(Dead Letter Queue) | 재학습 실패 시 메시지가 DLQ에 보존 → 수동 재처리 가능 |
| 재시도 내성 | Worker가 죽어도 SQS visibility timeout 후 메시지 재노출 → 자동 재시도 |

**SNS 대신 SQS를 선택한 이유**: SNS는 Pub/Sub으로 여러 구독자에게 동시 전달이 목적이다. 재학습 작업은 단일 Worker가 순서대로 처리해야 하므로 큐(SQS)가 적합하다. SQS는 메시지를 안전하게 보관하고 하나의 소비자가 처리를 완료한 후 삭제하는 at-least-once 보장을 제공한다.

### 2.5 왜 API Gateway인가

#### 직접 ALB 노출의 문제점

ALB(Application Load Balancer)를 인터넷에 직접 노출하면 다음 문제가 생긴다.

- Rate Limiting: ALB 자체에는 요청 속도 제한 기능이 없다.
- API Key 관리: API Gateway Usage Plan으로 키별 할당량을 설정할 수 없다.
- 인증: JWT·API Key 검증을 서버 애플리케이션이 직접 처리해야 한다.

#### API Gateway를 선택한 이유와 효과

| 특성 | 효과 |
|------|------|
| Usage Plan + API Key | 키별로 요청 할당량(quota) 및 속도 제한(throttle) 설정 |
| 인증 오프로드 | Lambda Authorizer 또는 Cognito를 붙이면 서버 코드 수정 없이 인증 추가 |
| CloudWatch 자동 연동 | 4xx/5xx 카운트, 지연시간 메트릭을 별도 구성 없이 수집 |
| HTTPS 기본 제공 | SSL 인증서 관리 불필요 |

## 4. 측정 결과 비교

### 4.1 Baseline (10 VU, 5분)

| 지표 | Stage 1 | Stage 2 | Stage 3 (0.5 vCPU) | Stage 3B (4 vCPU) |
|------|---------|---------|--------------------|-------------------|
| p50 latency | 818 ms | **255 ms** | 1,711 ms | **59 ms** |
| p95 latency | 2,979 ms | 2,254 ms | 2,407 ms | **97 ms** |
| p99 latency | 8,074 ms | 4,182 ms | 2,660 ms | - |
| HTTP 실패율 | 0.7% | **0.0%** | **0.0%** | **0.0%** |
| RPS | 4.61 | **6.41** | 3.76 | **9.37** |
| 통과 기준 (p95<500ms) | FAIL | FAIL | FAIL (0.5vCPU 병목) | **PASS** |

> **Stage 3 p50이 Stage 2보다 높은 이유**: ECS 0.5 vCPU 제한 + API Gateway → ALB → ECS 네트워크 홉(~50~100ms) 추가. 단, p99는 2,660ms로 Stage 2(4,182ms)보다 36% 개선 — 클라우드 인프라의 안정성이 극단적 지연을 줄인 것.

### 4.2 Spike (0→100 VU, 3분)

| 지표 | Stage 1 | Stage 2 | Stage 3 (0.5 vCPU) | Stage 3B (4 vCPU) |
|------|---------|---------|--------------------|-------------------|
| p95 latency (피크) | 15,016 ms | 15,005 ms | 15,001 ms | **3,345 ms** |
| 오류율 (피크) | 81.9% | 67.3% | 93.3% | **0.0%** |
| HTTP 실패 수 | 715 | 696 | 816 | **0** |
| max latency | - | - | 15,001 ms | **4,010 ms** |
| 통과 기준 (p95<2000ms) | FAIL | FAIL | FAIL (CPU 한계) | FAIL (단일 태스크 한계) |

> **모든 Stage에서 p95가 ~15,001ms에 수렴하는 이유**: k6 클라이언트 timeout(15초) 값이다. 서버가 응답을 보내지 못하는 것이 아니라, 100 VU 동시 요청이 CPU 처리 큐에 쌓여 대기하다가 클라이언트가 먼저 timeout으로 끊는 것이다.
> **Stage 3B FAIL 원인**: 4 vCPU여도 단일 ECS 태스크 1개가 100 VU를 처리. `desired_count=2`로 태스크 수를 늘리면 해결 가능.
