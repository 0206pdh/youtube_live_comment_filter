## Baseline (10 VU, 5분)

| 지표 | Stage 1 | Stage 2 | 통과 기준 |
|------|---------|---------|----------|
| 총 요청 수 | 1,384 | 1,922 | - |
| p50 latency | 818 ms | **255 ms** | - |
| p95 latency | 2,979 ms | 2,254 ms | < 500ms |
| p99 latency | 8,074 ms | 4,182 ms | - |
| RPS | 4.61 | **6.41** | - |
| HTTP 실패율 | 0.7% | **0.0%** | < 1% |
| 통과 여부 | FAIL (p95 초과) | FAIL (p95 초과) | - |

> **해석**: Stage 2가 2개 컨테이너로 분산처리하여 p50 latency가 818ms → 255ms로 68% 개선,
> RPS도 4.61 → 6.41로 39% 향상. 그러나 CPU BERT 특성상 p95는 여전히 기준 초과.

## Spike (0→100 VU, 3분)

| 지표 | Stage 1 | Stage 2 | 통과 기준 |
|------|---------|---------|----------|
| 총 요청 수 | 873 | 1,033 | - |
| p50 latency | 14,999 ms | 14,998 ms | - |
| p95 latency | 15,016 ms | 15,005 ms | < 2,000ms |
| HTTP 실패 수 | 715 | 696 | - |
| 오류율 | 81.9% | 67.3% | < 5% |
| 통과 여부 | FAIL | FAIL | - |

> **해석**: 100 VU 스파이크 시 두 환경 모두 15초 타임아웃에 수렴.
> Stage 2가 컨테이너 2개라 실패 수(715→696)와 처리량(873→1,033)에서 소폭 개선됐으나,
> CPU BERT 단일 쓰레드 한계로 근본적 해결 불가.

## 데이터 일관성 (Stage 2 전용)

| 지표 | 값 |
|------|-----|
| 총 조회 횟수 | 1,656회 |
| 일치 (hit) | **0회** |
| 불일치 (miss) | **1,656회** |
| **불일치율** | **100.0%** |

> **해석**: nginx 라운드로빈이 저장 요청을 server_a 또는 server_b로 무작위 분산.
> 조회 시 다른 서버가 응답하면 데이터가 없음 → 불일치율 100%.
> 이는 로컬 파일시스템 기반 스케일아웃의 근본적 문제를 수치로 증명.
> ECS Fargate + S3/RDS 환경에서는 이 수치가 0%가 되어야 함.

## 결론: Stage 3 (ECS Fargate + S3/RDS) 필요 이유

1. **성능**: CPU BERT는 동시 요청 처리 불가 → ECS Fargate에서 task 단위 병렬 처리 필요
2. **데이터 일관성**: 로컬 파일 기반은 스케일아웃 시 100% 불일치 → S3/RDS로 중앙화 필수
3. **내구성**: 컨테이너 재시작 시 user_data/ 소실 → S3에 영구 저장 필요
