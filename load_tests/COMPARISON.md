## 결론

### Stage 1 문제점
- 단일 인스턴스 한계: 트래픽 급증 시 처리 불가
- 컨테이너 재시작 시 `user_data/` 소실 → 학습 데이터 영구 손실

### Stage 2 문제점
- 스케일 아웃은 가능하지만 각 인스턴스가 다른 데이터를 가짐
- 불일치율 **100%** 측정 (1,845건 중 1,845건 miss) → 동일한 요청이 어느 서버에 닿느냐에 따라 다른 결과 반환
- 실제 서비스에서는 사용자가 제출한 피드백이 일부 서버에만 저장됨

### Stage 3 선택 이유 (공통)
- **ECS Fargate**: 컨테이너 인프라 관리 불필요, 태스크 단위 독립 스케일 아웃
- **S3**: 모든 태스크가 동일한 버킷 접근 → 데이터 불일치율 0%
- **RDS**: 트랜잭션 보장, 여러 태스크에서 동시 쓰기 시 일관성 유지

### Stage 3 Phase 3 (Worker 분리) 핵심 효과
- SQS + Worker ECS로 재학습을 API와 완전 분리 → 재학습 중 추론 latency 영향 없음
- **같은 4 vCPU**: Stage 1 p95=866ms, 오류율 49.93% → Phase 3B p95=107ms, 오류율 0% (구조 개선만으로 달성)
- 0.5 vCPU 단독으로도 재학습 동시 시나리오 SLO 통과 가능 (Worker 분리 시)

### Stage 3B (4 vCPU) 결과
- Baseline p95: 2,407ms → **97ms** (-95.8%) → **SLO PASS**
- Soak p95: 9,469ms → **431ms** (-95.4%) → **SLO PASS**
- Spike max: 15,001ms → **4,010ms** (-73.3%), 타임아웃 완전 해소
- HTTP 오류율 Soak: 3.0% → **0.004%**, Spike: 93.3% → **0.0%**
