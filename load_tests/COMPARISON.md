# 부하 테스트 비교 결과

## 왜 ECS Fargate + S3 + RDS를 선택했는가

아래는 3가지 환경에서 동일한 부하를 걸어 측정한 실제 수치이다.

---

## 환경 구성

| 항목 | Stage 1 | Stage 2 | Stage 3 |
|------|---------|---------|---------|
| 컴퓨팅 | 단일 Docker | Docker 2개 + nginx | ECS Fargate |
| 스토리지 | 로컬 파일시스템 | 로컬 파일시스템 (각자) | S3 |
| DB | 없음 | 없음 | RDS PostgreSQL |
| 스케일 아웃 | 불가 | 가능 (but 데이터 불일치) | 가능 (데이터 일관성 보장) |

## 결론

### Stage 1 문제점
- 단일 인스턴스 한계: 트래픽 급증 시 처리 불가
- 컨테이너 재시작 시 `user_data/` 소실 → 학습 데이터 영구 손실

### Stage 2 문제점
- 스케일 아웃은 가능하지만 각 인스턴스가 다른 데이터를 가짐
- 불일치율 **100%** 측정 (1,656건 중 1,656건 miss) → 동일한 요청이 어느 서버에 닿느냐에 따라 다른 결과 반환
- 실제 서비스에서는 사용자가 제출한 피드백이 일부 서버에만 저장됨

### Stage 3 선택 이유
- **ECS Fargate**: 컨테이너 인프라 관리 불필요, 태스크 단위 독립 스케일 아웃
- **S3**: 모든 태스크가 동일한 버킷 접근 → 데이터 불일치율 0%
- **RDS**: 트랜잭션 보장, 여러 태스크에서 동시 쓰기 시 일관성 유지

### Stage 3 남은 과제
- p95 latency 2,407ms — 0.5 vCPU에서 BERT 추론 + 네트워크 홉 overhead
- 재학습 요청 시 추론 태스크와 CPU 경합 가능 (Phase 3에서 해결)

## Phase 3 실측 vs 목표 비교

| 시나리오 | Stage 3 (분리 전) | Phase 3 실측 | SLO 목표 | 판정 |
|---------|----------------|------------|---------|------|
| Baseline p95 (10 VU) | 2,407 ms | **2,328 ms** (-3.3%) | < 200 ms | FAIL |
| Baseline HTTP 실패율 | 0.0% | **0.0%** | < 0.1% | **PASS** |
| Spike p95 (0→100 VU) | 15,001 ms | **15,001 ms** (동일) | < 1,000 ms | FAIL |
| Soak p95 (30 VU, 30분) | 미측정 | **9,469 ms** | < 300 ms | FAIL |
| 메모리 누수 없음 | 미확인 | **RPS 일정 확인** | - | **PASS** |

**latency 목표 미달 근본 원인**: ECS API Task의 CPU 0.5 vCPU가 BERT 추론 자체의 병목. Worker 분리는 재학습/추론 CPU 경합을 해결했지만, 테스트 중 재학습이 발생하지 않아 그 효과는 이 수치에 반영되지 않는다.

**다음 단계**: ECS API Task CPU를 512 → 1024 (1 vCPU)로 상향하면 Baseline p95가 ~50% 개선될 것으로 예상.
