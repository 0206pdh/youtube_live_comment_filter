## 결론

### Stage 1 문제점
- 단일 인스턴스 한계: 트래픽 급증 시 처리 불가
- 컨테이너 재시작 시 `user_data/` 소실 → 학습 데이터 영구 손실

### Stage 2 문제점
- 스케일 아웃은 가능하지만 각 인스턴스가 다른 데이터를 가짐
- 불일치율 **100%** 측정 (1,656건 중 1,656건 miss) → 동일한 요청이 어느 서버에 닿느냐에 따라 다른 결과 반환
- 실제 서비스에서는 사용자가 제출한 피드백이 일부 서버에만 저장됨

### Stage 3 선택 이유
- **ECS Fargate**: 컨테이너 인프라 관리 불필요, 태스크 단위 독립 스케일 아웃
- **S3**: 모든 태스크가 동일한 버킷 접근 → 데이터 불일치율 0%
- **RDS**: 트랜잭션 보장, 여러 태스크에서 동시 쓰기 시 일관성 유지
- Soak 테스트에서 30분간 latency 증가 없음 → 메모리 누수 없음
